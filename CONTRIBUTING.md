# Contributing to the CyberSkill Design System

*Read `docs/00-audit-and-roadmap.md` once before contributing — it is the operating guide for the doctrine, the audit framework, and the change pipeline. This file is the on-ramp; the operating guide is the source of truth.*

---

## Who can contribute

Anyone — internal CyberSkill engineers/designers, external consumers, and AI agents. The bar isn't *who*; the bar is *which kind of change* and *which review track it takes*.

## Three contribution tracks

### 1. Editorial — typo / link / formatting / cross-reference repair

Fastest track. No RFC required. PR review by the chair owner of the affected part (5-day SLA). Ships as a **patch** version (per `00-audit-and-roadmap.md` §12.1).

**Examples:** fixing a typo in `part-3a-actions.md`; repairing a broken link to `part-7-engineering-operations.md`; updating a date in a reference table.

### 2. Substantive — new section, new pattern, new component, expansion to an existing part

Requires a **Doctrine RFC** filed at `docs/RFCs/{YYYY}-{NNN}-{slug}.md` per the §12.3 template. 14-day review window. Ships as a **minor** version (`x.Y.0`).

**Examples:** new RFC pattern in `part-11-enterprise-patterns.md`; new vertical pack in `part-19-vertical-packs.md`; a new headless primitive in `@cyberskill/primitives`; a new theme overlay scheme.

### 3. Breaking — structural reorganisation, removal of a documented contract, renumbering

Requires a Doctrine RFC + Founder approval + 30-day review window. Ships as a **major** version (`X.0.0`).

**Examples:** renaming a token slot; removing a primitive variant; restructuring section numbering. **You should think hard before filing one of these — the v1.0 lock exists for a reason.**

## Hard constraints (`00-audit-and-roadmap.md` §13.2)

These are non-negotiable. PRs that violate them will be closed.

1. Never modify the brand anchors (Umber, Ochre, slogan, fonts) — Part 1 §2 immutable.
2. Never delete or renumber any `part-*.md` file. Numbers 1–20 are stable identifiers.
3. Never silently translate customer-facing or legal-binding text.
4. Never invent regulatory citations. If unsure, write "TBD — needs legal review".
5. Never modify the Vietnamese-first commitment.
6. Never bypass the change pipeline.
7. Never produce content violating Part 1 §3 voice or Part 14 §2.3 banned phrases.
8. Never fabricate test results, telemetry data, adoption metrics, or audit scores.
9. Never edit `DESIGN.md` by hand — it's generated by `scripts/build-design-md.mjs`.

## Step-by-step — opening a PR

1. **Decide your track** (Editorial / Substantive / Breaking) per above.
2. **Read the affected part(s)** in full + every part it cross-references. If you're an AI agent, see `00-audit-and-roadmap.md` §13.9 session checklist.
3. **For Substantive / Breaking**: file a Doctrine RFC at `docs/RFCs/{YYYY}-{NNN}-{slug}.md` per the §12.3 template. Use the next sequential `NNN`. Add a row to `docs/RFCs/_index.md` with status `Draft`.
4. **For Editorial**: skip step 3.
5. **Make the change**:
   - Match the existing voice (Part 1 §3 — warm / direct / honest / respectful).
   - Use canonical phrases (Part 14 §2.6 glossary).
   - Reference tokens, not literals (`color.semantic.danger` not `#B33B19`).
   - Update Vietnamese counterpart in the same PR or note `vi-deferred-by:` per Part 5 §7.
6. **Run the local checks** before opening the PR:
   ```bash
   pnpm typecheck
   pnpm lint
   pnpm pre-review            # RFC 2026-005 — token-discipline + banned-phrase + xref + a11y note
   pnpm coverage:check        # adoption metrics
   pnpm check:bundle-size     # per-package budgets
   pnpm check:doc-freshness   # broken-xref walk
   pnpm check:design-md       # DESIGN.md staleness
   pnpm build:tokens          # multi-platform token build
   ```
   The `pnpm ci` super-script runs all of these.
7. **Open the PR**. Title format: `[area] short imperative summary`. Reference the RFC if applicable.
8. **AI-agent PRs**: use the §13.4 PR template. The CI's pre-review job will post a comment summarising findings; address Block-class findings before requesting review.

## RFC template

See `docs/00-audit-and-roadmap.md` §12.3 for the full template. Required fields:

- Author, Affected parts, Class, Subtype, Status
- Audit reference (link to the criterion this closes; or "novel")
- Review window opens / closes
- Motivation, Proposed change, Alternatives considered (≥ 2 incl. "do nothing")
- Impact on dependent parts, Backward compatibility, Translation impact
- A11y impact, Telemetry impact, Audit-score impact, Tree-test/card-sort impact
- DESIGN.md impact, Open questions, Approver

The MCP server's `draft_rfc` tool (run with `--enable-write`) generates a stub for you — see `packages/mcp-server/`.

## Reviewers

- **Editorial**: chair owner of the affected part (per `00-index.md` ownership matrix).
- **Substantive**: chair owner + cross-referenced part owners + Founder.
- **Breaking**: chair owner + Founder + 30-day public review.

## AI-agent contribution tiers

Per `00-audit-and-roadmap.md` §13.1, AI agents operate under one of:

| Tier | What's allowed |
|---|---|
| A0 | Read-only; quote, regenerate `DESIGN.md`, draft an audit (Draft status only) |
| A1 | Editorial proposal via PR — chair-owner reviews |
| A2 | Substantive proposal via Doctrine RFC |
| A3 | Autonomous merge — not currently granted |

AI-agent PRs use the §13.4 PR template and pass the §13.5 review gates (voice rubric, citation, length, banned-phrase, cross-reference, translation, audit-score impact, DESIGN.md regeneration).
