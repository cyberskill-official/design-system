# Template content-schema v2

Purely additive, opt-in metadata: a sidecar `templates/<slug>/content-schema.json` next to a template's `.dc.html`, declaring the **type** of each `{{ hole }}` already in the template — so an agent or pipeline can populate/validate client content without parsing HTML.

**Zero template-markup changes required.** The slot `id` is the hole name already used in the `.dc.html`.

## Shape

```json
{
  "$schemaVersion": "2.0",
  "template": "marketing-page",
  "slots": [
    { "id": "heroH", "type": "text", "label": "Hero headline", "maxLength": 70, "required": true },
    { "id": "heroP", "type": "richtext", "label": "Hero paragraph" }
  ]
}
```

| Field | Required | Notes |
|---|---|---|
| `$schemaVersion` | yes | `"2.0"` |
| `template` | yes | matches `templates/<slug>/` |
| `slots[].id` | yes | equals a real `{{ id }}` hole |
| `slots[].type` | yes | `text` · `richtext` · `image` · `link` · `list` · `table` · `date` |
| `slots[].label` | yes | human-readable |
| `slots[].maxLength` | no | soft authoring guide |
| `slots[].required` | no | default `false` |
| `slots[].i18n` | no | `{ "en", "vi" }` for bilingual templates |

Machine-checkable shape: `templates/schema/content-schema.schema.json`. Gate: `_audit/template-schema-test.html` (every hole-driven sidecar). **27** opt-in sidecars ship today — high-traffic product surfaces keep growing measurable content-slot coverage; axis holes stay out of schema.
